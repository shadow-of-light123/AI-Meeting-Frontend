## 五、STAR 法则详细版（适用简历"项目经历"主体）

### S - 背景

> 该项目是一个基于大语言模型的 AI 模拟面试与智能对话 SaaS 应用，核心功能包括：简历解析与智能出题、多轮问答与追问、语音转写与 TTS 播报、摄像头神态分析、面试报告与雷达图。后端为 Spring Boot 3 单体架构，前端由本人独立负责架构设计与全量开发。

### T - 任务

> 需要解决的核心问题：
>
> 1. 多种实时异步场景下的并发竞态（SSE 流取消、WebSocket 热启停、会话创建与消息发送的时序依赖、摄像头轮询堆积）
> 2. AI 流式响应的逐字渲染体验与多协议兼容
> 3. 音频全链路的前端工程化（采集→编码推流→转写→去重→展示）
> 4. 后端 API 多版本 Schema 漂移的无感兼容

### A - 行动

> **1. 自研请求治理引擎**（`src/lib/request.ts`, 650 行）
>
> - 设计 join / cancel-previous / reject / debounce 四种策略，覆盖幂等读、防重复提交、竞态覆盖等场景
> - 实现稳定的请求 Key 序列化器，处理循环引用、FormData、有序对象等边界
> - 批量消费者模式的 debounce：多个并发调用者共享单次执行结果
> - AbortSignal 跨层传播，确保上层取消信号穿透去重层
>
> **2. 竞态治理体系**
>
> - Pending Outbound 延迟执行模式，解决"会话创建中再次发送消息"的时序依赖
> - Symbol 令牌机制，解决录音快速启停时的异步回调误执行
> - `activeRequestIdRef` 门禁 + `AbortController`，三层取消协议保证 SSE 回调安全
> - `isUploadingRef` 互斥锁防止摄像头轮询上传堆积
>
> **3. SSE 流式渲染系统**
>
> - 基于 `@microsoft/fetch-event-source` 实现可靠的 SSE 连接
> - 自研 `TextStreamLimiter` 逐字输出控制器（惰性启停、按需刷新、双通道并行）
> - 兼容 OpenAI 兼容格式与自定义 JSON 格式，检测 10+ 种流结束信号
> - 客户端模拟流式：对同步返回的长文本做分帧渲染，可取消、可中断
>
> **4. WebSocket 音频管线**
>
> - 连接态缓冲队列（`pendingBinaryQueue`），解决 WebSocket 握手期间音频数据丢失
> - 时间戳光标 + 内容指纹双重去重，防止重连导致重复转录
> - 实时 PCM16 编码、15s 心跳保活、运行时协议自动检测（ws/wss）
> - Float32 → Int16Array 采样转换与 WebKit AudioContext 兼容
>
> **5. 工程化与体验**
>
> - Redux Toolkit（客户端状态） + TanStack Query（服务端缓存）双层状态管理，`authEpoch` 全局失效信号
> - 五级错误码命名空间 + `AppError.from()` 智能转换器，消除 try-catch 样板代码
> - Vite `manualChunks` 按业务域分包（PDF 预览/状态管理/UI 库），优化缓存命中率
> - API 路径 404 自动降级 + 字段名多版本标准化，实现后端 Schema 漂移的无感兼容
> - 12 个测试模块覆盖核心工具库、状态机和 Hook

### R - 成果

> - 前端项目完全独立交付，代码量 15000+ 行（不含 node_modules）
> - 实现了覆盖 HTTP/SSE/WebSocket/WebRTC(摄像头)/Web Audio 的全链路异步治理
> - 请求去重引擎可作为独立 npm 包复用
> - 面试全流程（简历→出题→多轮问答→神态分析→报告）线上稳定运行

## 七、面试中可能被追问的问题（提前准备）

| 问题                                                             | 准备要点                                                                                                                                                               |
| ---------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| "请求去重的 stableSerialize 遇到循环引用怎么办？"                | `WeakSet` 记录已遍历对象，第二次遇到跳过                                                                                                                               |
| "为什么 GET 默认 join，POST 默认 off？"                          | HTTP 语义：GET 幂等可合并，POST 非幂等不可合并                                                                                                                         |
| "Symbol 令牌和 boolean flag 的区别？"                            | Symbol 每次创建唯一，避免多个异步操作共用一个 boolean 的误判                                                                                                           |
| "WebSocket 连接态缓冲是什么，为什么需要？"                       | 用户按下录音按钮瞬间就开始说话，WS 还在握手，这个窗口期的音频数据不能丢                                                                                                |
| "两个独立的流式渲染（SSE + 模拟）为什么不统一？"                 | SSE 流是增量到达的，不需要客户端分帧；模拟流是一次性拿到的完整文本，需要客户端拆分渲染。场景不同，没法统一                                                             |
| "Redux Toolkit 和 TanStack Query 分别管什么，为什么不只用一个？" | Redux 管客户端运行时状态（当前流状态、pending outbound），TanStack Query 管服务端缓存（会话列表、历史消息）。用 RTK Query 替代也行，但 TanStack Query 的缓存策略更灵活 |
| "如果用户量增大，SSE 连接数过多怎么办？"                         | 考虑使用共享 SSE 通道或降级为轮询；做好连接池管理和自动重连；评估是否需要用 WebSocket 替代 SSE                                                                         |
| "ScriptProcessorNode 已废弃，你打算怎么替换？"                   | 迁移到 AudioWorklet，在独立线程中处理音频，避免主线程阻塞。已经在调研中（或计划中）                                                                                    |

## 九、经验之谈：怎么写一个"能引起追问"的项目描述

1. **不要只写"做了什么功能"**——面试官看了无感。"实现了 AI 对话功能"❌ → "自研请求去重引擎，编排四种策略处理并发竞态"✅

2. **埋具体的数字和技术名词**——给面试官提供追问的锚点。"650 行"、"四种策略"、"Symbol 令牌"、"批量消费者模式"——这些词面试官一看就知道有东西可以问

3. **写你真正有深度参与的部分**——你要是简历上写了"请求去重引擎"，面试官问你 `stableSerialize` 怎么处理 `FormData` 你得能答上来。写上去的每一项都是你接下来要面对的考题

4. **诚实写不足比编造更可信**——如果被问到"这个项目有什么不足"，直接说"缺 E2E 测试、ScriptProcessorNode 待迁移到 AudioWorklet、部分流解析逻辑有复制"，比说"我觉得都挺好的"可信十倍

5. **按目标岗位调整侧重点**——投 AI 方向突出 SSE 和流式渲染，投音视频方向突出 WebSocket 和 Web Audio，投全栈突出独立交付和后端协作。同一个项目，不同侧重点
